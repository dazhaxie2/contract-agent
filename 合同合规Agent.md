假设你是AI应用开发资深架构师，需要你来给合同合规场景的Agent系统设置RAG架构，面对上万的初始法律、合同、合规等长文档，需要批量入关系数据库、向量数据库、知识图谱库、文件存储库，需要抗住1000QPS的并发量，你会从哪些方面去设计这套系统。比如Rag选什么前沿架构，从文档预处理、分块、索引构建策略、入库策略、检索策略、模型选择策略、排序策略、大模型生成策略、提示词策略、上下文管理策略、记忆策略、Agent框架、抗高并发策略、高可用策略、高扩展性策略、可视化链路追踪策略、召回质量评估体系等方面去设计



思考链：

为设计合同合规场景 RAG Agent 系统，我需锚定核心痛点。该场景要求支撑上万长文档、1000QPS 并发，覆盖全链路。其痛点在于：合同合规专业性强、准确性要求高，不能出现幻觉；长文档数量多，如法律条文、合同模板等；并发要求高，可能是集团法务等多人同时使用或对外提供服务；还涉及多库协同，包括关系、向量、知识图谱及文件存储。



为设计合同合规场景 RAG Agent 系统，我需先确定整体架构选型。考虑到合同合规专业性强、准确性要求高，且长文档数量多，我决定采用多阶段分层 Agentic RAG 架构，融合 Graph RAG、Self-RAG、CRAG、Multi-Index RAG。其中，Graph RAG 能处理实体和关系检索，解决语义断层问题，以贴合合同合规场景强逻辑关联的需求。



为设计合同合规场景 RAG Agent 系统，我会进一步细化架构与预处理策略。Self-RAG 和 CRAG 可解决幻觉问题，需建立检索 - 校验等闭环。Multi-Index 多索引架构对应多库，不同内容用不同检索方式。Agent-native 能拆解多任务。文档预处理与分块很关键，上万长文档有结构，需精准预处理。我计划采用预处理流水线，解析多种文档格式，保留结构，过滤重复页眉页脚。



我会对合同合规场景的文档进行格式归一化与清洗，包括去重、去噪、格式归一、提取元数据。元数据对区分有效和失效法律至关重要，将用小模型提取。在分块策略上，采用语义、结构、层级结合的混合分块，以法律条款 “条 / 款” 为最小原子分块，保留权责关系。



为解决合同合规场景长文档检索上下文丢失问题，我计划采用层级分块，以 “章、节、条、款” 构建父块 - 子块结构，赋予每个块唯一层级 ID，实现精准检索与上下文回溯。对于无明确结构的文档，用语义分块，控制块大小在 128 - 1024token，超大块做摘要分块并保留原文块。同时，增强分块，为每个块附加元数据及合规要点摘要，提升召回率。此外，对表格和定义条款单独分块，分别存入结构化文本、表格语义摘要及知识图谱。



为满足合同合规场景上万文档的多库协同需求，我计划先确定四库选型与索引策略。文件存储库存原始及结构化文件，可选 MinIO 等，用于归档、版本管理等，以 MD5 和元数据索引存于关系库。关系型数据库存结构化元数据等，1000QPS 高并发场景，可选 PostgreSQL 或 TiDB，PostgreSQL 支持 JSONB 且可与向量库联动，TiDB 适合分布式高并发。同时，设计批量、异步、可重试、幂等的入库流水线。



我将设计合同合规场景核心表，包括文档表、文档块表等，还会设计用户表等辅助表。构建主键、联合、全文及前缀索引。采用批量、异步、幂等的入库策略，单批次 1000 - 5000 条，先写文件库，再写关系库，最后写向量库和图谱库。



为满足合同合规场景需求，向量数据库需支撑 1000QPS，应对上万文档及千万级向量扩展。选型上，开源有 Milvus、Qdrant，商用有 Pinecone，私有化选 Milvus 和 Qdrant 更合适。嵌入模型方面，因法律术语语义特殊，必须用法律领域微调模型。开源可选 bge-large-zh-v1.5 等，商用有通义千问等，私有化需选开源可部署的。



为满足合同合规场景需求，我会从向量维度、索引构建策略等方面规划。向量维度设为 1024 维，平衡精度与性能。索引构建策略上，百万级向量用 IVF_FLAT，千万级用 HNSW，其 QPS 更高、延迟更低，参数设为 M=16, ef_construction=200，ef_search=64。还会按文档类型、生效日期等做分区索引，对元数据做标量过滤索引，每个块存原文及摘要向量做多向量索引，以提升检索效率与召回率。



我计划采用批量异步入库策略，单批次 500 - 1000 条，与关系库联动，写入成功后异步生成向量写入向量库，支持断点续传与自动重试，利用 chunk_id 实现幂等写入。同时，实行冷热分离，将热数据存于内存节点，冷数据存于磁盘节点，还会隔离写入与查询节点，避免影响在线 QPS。知识图谱库用于解决逻辑推理问题，选型上，高并发场景优先考虑开源的 NebulaGraph，其分布式、高并发，能满足 1000QPS 需求，而 Neo4j 单机强，分布式企业版成本高。



为构建合同合规领域知识图谱，我将从本体设计、实体与关系抽取、索引构建三方面入手。本体设计定义实体类型如法律主体等，关系类型如引用等；实体与关系抽取用法律微调模型批量处理文档块提取；索引构建有点索引和边索引，分别提升实体与关系查询速度。



我将对实体描述及属性做全文索引，支持关键词检索实体。入库策略上，采用批量写入，单批次 1000 - 5000 条，先点后边，保证原子性；去重合并相同实体；增量更新新文档；读写分离，避免写入影响查询。整体采用分布式异步 ETL 流水线，如用 Airflow 调度或 Kafka + Flink 流处理，经上传、解析预处理阶段，生成结构化文档和块存入关系库。



我计划在向量生成和图谱构建阶段，分别调用嵌入模型和实体关系抽取服务，将生成的向量和抽取的实体关系写入对应数据库。同时，全链路进行监控与重试，用 doc_id+chunk_id 保证幂等性。在检索策略上，采用混合、多轮、分层检索融合策略，结合 Self-RAG 和 CRAG 思想，先对用户 Query 进行预处理，提取意图、实体及过滤条件。



我计划在检索前对 Query 进行改写和合规校验。改写用小模型将口语化 Query 转为法律专业术语，并生成多个子 Query 以提升召回率；合规校验判断 Query 是否在合规场景，避免无效检索。在混合检索阶段，分别用改写后的子 Query 做向量检索，Top-K=50，结合标量过滤；用关系库全文索引做关键词检索，Top-K=30，用 BM25 算法排序；抽取 Query 实体在知识图谱做检索，Top-K=20 。



为满足 1000QPS 低延迟需求，我计划让三个检索通道并行执行。在召回结果融合与粗排阶段，先按 chunk_id 去重并记录来源，再用轻量级排序模型对 Top-100 结果粗排至 Top-30，同时过滤失效文档。精排与校验阶段，用效果更好的重排模型对 Top-30 精排至 Top-10，通过小模型进行相关性和事实性校验，过滤不相关及无效内容。



若校验发现召回内容不足，触发二次检索，模型自行判断是否补充检索，重新改写 Query，直到召回足够有效内容或达最大检索次数。在上下文窗口构建阶段，对精排后的 Top-10 块回溯父块及文档元数据，补充上下文；严格控制上下文 token 数，按相关性、法律效力优先级排序，避免溢出。



我会对引用标注做出要求，每个内容块标注来源，方便溯源及模型生成引用。在模型选择上，采用大模型与小模型混合的分层架构，小模型负责前置处理等，大模型专注生成与复杂推理。分模块选型，嵌入模型选法律微调的 bge-large-zh-v1.5 等，预处理 / 小模型选 Llama 3 8B 法律微调版等体积小、推理快的模型，以满足 1000QPS 高并发需求。



我会让小模型处理无需复杂推理的任务，如 Query 改写等，解放大模型用于生成。重排模型方面，粗排选 BGE-reranker-base 法律微调版，精排选 BGE-reranker-large 法律微调版。核心生成大模型，私有化部署可选开源的 Llama 3 70B 等，商用 API 有通义千问法律大模型等，需根据实际情况选择。



为满足合同合规场景需求，我需明确大模型部署与策略要求。上下文窗口至少 32K，最好 128K；模型必须经合同合规、法律条文微调；大模型要支持函数调用；需用推理优化框架提升吞吐、降低延迟，1000QPS 要分布式部署多实例并负载均衡。同时，要严格设计提示词和生成策略，保证生成内容专业、准确、合规。



我将提示词分为系统提示词、任务提示词。系统提示词明确角色为资深合同合规法律专家，需以事实为依据、法律为准绳，标注引用来源，区分强制性与任意性规定，提示风险，确保内容结构清晰。任务提示词针对合同审查等不同 Agent 任务，明确任务目标、输出格式与校验规则。



我会针对合同审查等任务设计提示词，如要求对合同合规性评价、风险点标注及整体修改方案输出，且要有引用依据。同时，动态注入上下文、对话历史等内容，调整 token 分配。生成策略上，结合 Self-RAG 和 CRAG 思想，采用生成 - 校验 - 修正闭环，约束大模型闭卷生成，防止幻觉。



为确保合同合规场景生成内容准确，我计划采用分步生成方式，如合同审查先生成风险点清单，再依据及建议，每步校验。生成后用小模型做事实性校验，检查检索依据、法律冲突等，有误则修正或补充检索。同时做合规性校验，拦截违规建议。还需实现引用溯源，点击引用能跳转原始文档。此外，将生成温度设为 0.1 - 0.3，保证内容稳定。





为满足合同合规场景大模型上下文需求，我计划采取以下策略：

- 窗口分层管理，将上下文窗口按 2K、4K、16K、10K token 分别预留系统提示词、对话记忆、检索上下文、生成内容，总 token 数控制在 32K 内，128K 窗口模型可适当扩大检索上下文预留。
- 动态压缩与优先级排序，按法律条文、合同核心条款等优先级对检索内容排序，优先保留高优先级内容，对低优先级内容用小模型摘要压缩，超限时丢弃或压缩优先级最低内容。
- 语义去重，对检索内容做语义去重，减少 token 占用。
- 长文档处理，针对整份合同审查，采用分层滑动窗口与关键信息锚定策略，提取核心信息作锚点，分章节处理，避免文档溢出。



我将 Agent 记忆分为短期记忆与长期记忆。短期记忆即对话上下文记忆，用滑动窗口管理最近 5 - 10 轮对话，超量部分用小模型摘要压缩，还会提取核心实体等信息存入实体记忆。长期记忆包含用户画像、合规规则及历史任务记忆，分别用关系库、RAG 知识库存储，用户对话时自动注入相关信息，保证生成内容贴合需求、符合企业合规要求，并保持任务连贯性。



我将从记忆检索更新与隔离，以及 Agent 框架设计两方面规划。记忆方面，对话时检索并注入记忆，结束后提取核心信息增量更新，不同用户企业记忆隔离。Agent 框架基于 Agentic RAG 思想，采用 ReAct 框架与多智能体协同架构，主 Agent 负责任务拆解调度等，子 Agent 负责专项任务。



我计划让大模型具备任务拆解等核心能力，用 70B 级法律大模型支持复杂调度。设置文档解析等多种子 Agent，各有专属提示词、工具和模型，专注专项任务，确保专业性与准确性。



我计划将 Agent 可调用工具模块化、可扩展，包括检索、文档处理、合规校验、计算及外部接口工具。执行流程采用 ReAct 框架，通过 Thought、Action、Observation 循环，完成任务。同时，建立异常处理机制，子 Agent 执行失败自动重试，检索结果不足自动触发二次检索。



我计划在校验不通过时，自动触发修正，让子 Agent 重新生成或补充检索；任务超时时，自动终止并提示用户。在高并发支持方面，将每个 Agent 模块设计为无状态，支持水平扩展，部署多实例并负载均衡；任务执行采用异步队列，削峰填谷；对高频请求结果缓存，降低重复计算。此外，还需从架构、部署等多方面设计抗高并发策略，如全链路无状态设计，所有服务模块数据存于分布式存储，支持水平扩展。



我计划从服务拆分、读写分离、全链路异步化三方面优化系统架构。服务拆分将系统拆分为多个独立微服务，各服务独立部署扩展；读写分离使数据库、向量库、图谱库实现读写分离，提升查询 QPS；全链路异步化把非核心流程用消息队列异步处理，降低延迟。此外，还将构建多级缓存架构，一级缓存为本地内存缓存，缓存高频内容，过期时间 5 分钟；二级缓存是分布式缓存，缓存低频内容，过期时间 24 小时，支持集群部署。



我计划从缓存和流量控制两方面优化系统。在缓存方面，采用向量库冷热分离缓存，预热高频内容，设置防击穿、雪崩、穿透策略。流量控制上，用工具做全链路限流、熔断降级，通过消息队列削峰填谷，应对高并发。



在推理优化层面，大模型用 vLLM 等部署，支持多种技术提升 GPU 利用率；嵌入模型等用工具优化，批量推理提升吞吐；还会对模型实例分组。高可用策略上，基础设施层面多可用区部署与硬件冗余；服务层面待补充。



我计划从集群部署、服务健康检查、负载均衡三方面优化集群。集群部署用 K8s 编排，保证每个服务至少 3 个副本，故障时自动重启或新建实例；服务健康检查配置接口，K8s 定期检查，不健康实例自动下线；负载均衡用 SLB/Nginx 分发流量，故障实例自动剔除。数据层面，所有数据多副本存储，如文件、关系、向量、图谱库分别设置 3 副本；采用全量 + 增量备份，每天全量、每小时增量，异地存储；做跨地域容灾，主备集群分别在上海、北京，故障时快速切换，RPO<5 分钟，RTO<30 分钟。



在故障恢复层面，我计划采取多种措施。数据库、向量库、图谱库配置自动故障转移，主节点故障时从节点自动提升；所有版本更新做灰度发布，先少量发布验证，没问题再全量发布；每个版本保留回滚方案，出现问题可快速回滚；制定完善应急预案并定期演练。在高扩展性策略上，架构设计采用模块化、插件化，各模块可插拔，替换或新增无需改核心代码；接口标准化，新增模块实现标准接口即可接入系统。



我计划从领域、部署、数据、功能四方面增强系统扩展性。领域上，本体 Schema、提示词、任务流程支持扩展，新增领域只需添加对应本体等，无需改核心架构；部署采用云原生架构，基于 K8s 水平扩展，各服务独立扩缩容，还支持多种部署环境；数据层面，向量、图谱、关系库分布式部署，支持海量数据扩展，知识库更新增量更新；功能上，用户可开发自定义子 Agent 和工具接入，扩展系统功能。



我计划支持用户自定义提示词、任务流程及合规规则，无需修改系统代码。同时，为满足合同合规场景追溯需求，将搭建全链路追踪系统，基于 OpenTelemetry/Jaeger，给每个请求生成 Trace ID，各环节生成 Span，数据存入 Elasticsearch 等。还会设计可视化监控大盘，包括系统监控大盘展示整体状态，链路追踪大盘输入 Trace ID 可查看全链路执行过程。



我计划从三方面构建监控与分析系统。一是展示检索、大模型、Agent 执行情况的大盘，如检索质量大盘展示召回率等，方便优化策略；二是实现可观测性与告警，通过 ELK/Loki 采集全链路日志，Prometheus+Grafana 采集指标并设置阈值告警，自动分析慢链路、异常链路，给出优化建议。

# 合同合规场景高并发 Agentic RAG 系统架构设计

本架构针对**合同合规强专业性、零幻觉容忍、长文档密集、1000QPS 高并发**核心需求，采用**Agent-native 多模态混合 RAG 架构**，融合 Graph RAG、Self-RAG、CRAG 前沿技术，实现多库协同、全链路高可用、可扩展与可追溯，完整覆盖需求全维度。

## 一、核心 RAG 架构选型（前沿适配）

摒弃基础 Naive RAG，采用**分层 Agentic RAG 融合架构**，为合同合规场景量身定制：

1. 核心底座：Multi-Index Hybrid RAG + Graph RAG

   - 多索引混合检索：同时支持向量语义检索、全文关键词检索、知识图谱推理检索，解决纯向量检索的语义断层、逻辑关联缺失问题，适配法律条文引用、条款权责关联、合规规则推理等强逻辑场景。
   - 图谱增强：通过法律本体知识图谱，实现实体 - 关系的深度推理，解决 “条款违反哪条法律”“权责冲突识别” 等复杂合规问题，大幅提升召回准确率。

2. 纠错与自校验：Self-RAG + CRAG 双机制

   - Self-RAG：大模型自主判断是否需要检索、是否需要补充检索，避免无依据生成，从源头降低幻觉。
   - CRAG（Corrective RAG）：对召回内容做事实性、法律效力校验，过滤失效法条、错误内容，触发修正 / 二次检索，保证召回内容 100% 合规有效。

3. 任务调度：ReAct 多智能体协同架构

   

   主 Agent 负责任务拆解与调度，专项子 Agent 负责文档处理、合规审查、条款比对、法律检索等细分任务，实现复杂合同合规任务的自动化、流程化执行，适配多轮对话与长周期任务。

## 二、文档预处理与分块策略（RAG 效果根基）

针对法律合同的结构化特征（章 / 节 / 条 / 款 / 项 / 目），采用**结构优先 + 语义兜底的混合分块策略**，保证语义完整性与检索精准度。

### 1. 文档预处理流水线

1. 多格式精准解析

   - 可复制文档：用 PyMuPDF+LayoutLMv3 解析，完整保留章节目录、条款层级、表格、页眉页脚（自动过滤重复冗余内容）；
   - 扫描件 / 图片合同：用法律领域微调的 PaddleOCR 做文字识别 + 版面还原，支持公章、手写体、复杂表格解析；
   - 格式归一化：将所有文档转换为带层级的 Markdown/XML 结构，统一条款编号规则，保留完整的父子层级关系。

2. 清洗与元数据提取

   - 去重去噪：过滤水印、乱码、重复文档，自动识别并标注法条的有效 / 失效状态；
   - 元数据精准提取：用法律微调的 BERT 模型，提取文档类型（法律 / 合同 / 合规规范）、发布机构、生效 / 失效日期、适用行业、适用地域、版本号等核心元数据，为后续检索过滤提供基础。

3. 核心分块策略

   表格

   

   

   

   | 分块类型     | 适用场景                   | 实现规则                                                     |
   | ------------ | -------------------------- | ------------------------------------------------------------ |
   | 结构原子分块 | 正式法律条文、制式合同     | 以「条 / 款」为最小原子单元，完整保留单一条款的全部内容，不做强制 token 截断，避免权责语义断裂 |
   | 层级父子分块 | 长文档、多章节合同         | 构建「章→节→条→款」父子块树结构，每个子块关联父块 ID，检索时可回溯完整上下文，解决断章取义问题 |
   | 语义分块     | 无明确结构的合规指引、案例 | 用法律微调的 bge 分块模型做语义分割，块大小控制在 128-1024token，保证单块语义完整 |
   | 增强摘要分块 | 超长条款、复杂表格         | 对超大块生成合规要点摘要，同时保留原文块；表格转换为「结构化文本 + 语义摘要」单独分块，关联所属条款 |

   

4. 块增强处理

   

   每个块附加元数据标签、所属层级路径、法律效力优先级、实体标签，同时生成合规要点摘要，实现「原文 + 摘要」双向量索引，大幅提升召回率。

## 三、多库协同索引构建与入库策略

针对四大库（关系库、向量库、知识图谱库、文件存储库）做职责拆分与协同设计，支持上万长文档批量入库，兼顾写入效率与查询性能。

### 1. 四库选型与核心职责

表格







| 库类型       | 选型推荐                            | 核心职责                                                     | 高并发适配设计                                               |
| ------------ | ----------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 文件存储库   | MinIO（私有化）/ 阿里云 OSS（商用） | 原始文档归档、版本管理、权限管控、文件去重                   | 分布式多副本、S3 协议兼容、冷热数据分离、秒传去重            |
| 关系型数据库 | TiDB（分布式）/PostgreSQL           | 结构化元数据、块基础信息、用户权限、检索日志、Agent 执行链路 | 读写分离、分库分表、GIN 全文索引、联合索引优化、主从高可用   |

### 2. 索引构建核心策略

1. 关系库索引
   - 主键索引：doc_id、chunk_id、entity_id 等核心主键；
   - 联合索引：`doc_type + effective_date + status`（合规场景最高频过滤条件）；
   - 全文索引：对 chunk_content、chunk_summary 构建 GIN 全文索引，用于 BM25 关键词检索；
   - 前缀索引：对实体名称、条款编号构建前缀索引，加速精准查询。
2. 向量库索引
   - 主索引：采用 HNSW 索引（适配高并发低延迟场景），参数设置`M=16, ef_construction=200, ef_search=64`，平衡精度与性能；
   - 分区索引：按文档类型、生效状态、适用行业做分区，检索时仅扫描目标分区，大幅降低检索范围；
   - 标量过滤索引：对元数据字段构建标量索引，实现**先过滤、后检索**，避免全库扫描；
   - 多向量索引：每个块存储「原文向量 + 摘要向量」，双路检索融合召回。
3. 知识图谱索引
   - 本体 Schema 预定义：提前定义合同合规领域核心实体类型（法律主体、法条、条款、风险点、权责）与关系类型（引用、隶属、冲突、适用、定义）；
   - 点索引：对实体名称、实体类型构建索引，加速实体查询；
   - 边索引：对关系类型、出入度构建索引，加速关系推理与路径查询；
   - 全文索引：对实体描述、属性构建全文索引，支持关键词实体检索。

### 3. 批量入库流水线（分布式异步 ETL）

采用**Kafka+Flink+Airflow**构建全链路异步入库流水线，支持上万文档批量处理，保证幂等性与可重试性：

1. 上传归档：文档批量上传至文件存储库，生成唯一 doc_id，写入 Kafka 消息队列；
2. 解析预处理：Flink 消费消息，执行文档解析、清洗、元数据提取、分块，写入关系库，成功后推送至下一 Topic；
3. 向量生成入库：消费块数据，批量调用嵌入模型生成向量，异步写入向量库，单批次 500-1000 条，支持断点续传；
4. 图谱构建入库：消费块数据，调用法律微调的 NER / 关系抽取模型，抽取实体与关系，批量写入图谱库，先写点后写边，保证原子性；
5. 监控与重试：全链路埋点监控，失败消息进入死信队列自动重试，超阈值告警人工介入；
6. 读写隔离：入库操作走离线写入节点，与在线查询节点完全隔离，避免批量入库影响线上 QPS。

## 四、多阶段融合检索策略

针对合规场景高召回、高精准的要求，设计**前置处理→宽召回→粗排→精排→校验→二次检索**的多阶段检索流程，兼顾并发性能与召回质量。

1. Query 前置处理

   - 意图识别与实体抽取：用 7B 级法律微调小模型，识别用户 Query 的意图、核心实体、过滤条件（如行业、地域、法条生效时间）；
   - 多 Query 改写：将原 Query 改写为 3-5 个法律专业术语子 Query，覆盖不同检索维度，解决口语化、模糊 Query 的召回不足问题；
   - 合规拦截：过滤非合规场景请求，避免无效检索占用资源。

2. 并行宽召回（第一阶段）

   

   三个检索通道

   并行执行

   ，降低延迟，适配高并发：

   - 向量检索：多子 Query 并行检索，Top-K=50，余弦相似度匹配，前置标量过滤缩小检索范围；
   - 全文检索：基于关系库全文索引，用 BM25 算法做关键词精准匹配，Top-K=30，适配法律术语精准查询；
   - 图谱检索：抽取 Query 核心实体，做实体检索、路径检索、子图推理，召回关联的法条、条款、风险点，Top-K=20。

3. 粗排与过滤（第二阶段）

   - 去重合并：按 chunk_id 对三路召回结果去重，保留多来源标记；
   - 轻量粗排：用法律微调的 BGE-reranker-base 模型，对 Top-100 结果做粗排，筛选出 Top-30，保证速度的同时过滤低相关内容；
   - 合规过滤：剔除失效法条、不适用行业 / 地域的内容，保证召回内容的法律效力。

4. 精排与自校验（第三阶段）

   - 深度精排：用 BGE-reranker-large 法律微调版，对 Top-30 结果做精排，输出 Top-10 高相关内容；
   - 相关性校验：用小模型判断每个块与 Query 的相关性，过滤无关内容；
   - 事实性校验：校验内容的有效性、真实性，剔除与现行法律冲突的内容；
   - 二次检索触发：若校验后有效内容不足，自动触发 Self-RAG 机制，改写 Query 做补充检索，直到满足要求或达到最大检索次数。

5. 上下文窗口构建

   - 层级回溯：对精排后的块，回溯其父块、所属文档元数据，补充完整上下文，避免断章取义；
   - 优先级排序：按「法律条文→合同核心条款→合规规范→辅助内容」排序，保证高法律效力内容优先注入；
   - 引用标注：每个块标注来源（如《民法典》第五百七十七条、XX 合同第三章第五条），方便生成溯源与用户核对；
   - Token 管控：严格控制上下文 Token 占用，适配大模型窗口，避免溢出。

## 五、模型选择策略（大小模型分层架构）

采用**小模型做前置处理 / 检索 / 校验，大模型仅做最终生成 / 复杂推理**的分层架构，兼顾效果、成本与高并发支持。

1. 嵌入模型
   - 首选：**法律微调的 bge-large-zh-v1.5**，开源可私有化部署，中文法律语义适配性强，1024 维向量平衡精度与性能；
   - 商用备选：通义千问法律版嵌入模型、文心一言合规嵌入模型。
2. 轻量小模型（前置处理全链路）
   - 选型：Llama 3 8B/Qwen2 7B 法律合规微调版，开源可部署，推理速度快，延迟低；
   - 适用场景：Query 改写、实体 / 关系抽取、元数据提取、粗排、相关性校验、事实性校验、对话摘要，承接 90% 的预处理任务，大幅降低大模型压力。
3. 重排模型
   - 粗排：BGE-reranker-base 法律微调版，低延迟高吞吐；
   - 精排：BGE-reranker-large 法律微调版，高精准度，仅处理 Top-30 结果，性能压力可控。
4. 核心生成大模型
   - 私有化部署首选：Llama 3 70B/Qwen2 72B 法律合规微调版，支持 32K + 上下文窗口，适配长合同审查，用 vLLM 部署实现高吞吐；
   - 商用 API 备选：通义千问法律大模型、GPT-4o 法律版，成熟高并发支持，非涉密场景可用；
   - 核心要求：法律领域微调、支持函数调用、长上下文窗口、低延迟高吞吐。
5. 推理优化（高并发核心）
   - 所有模型用**vLLM/TensorRT-LLM**部署，支持 PagedAttention、连续批处理、动态批处理、INT4/FP8 量化，GPU 利用率提升 5-10 倍；
   - 模型实例分组：拆分高频请求组与复杂任务组，避免长文本生成占用全部资源，保障基础请求的 QPS。

## 六、生成与提示词策略（零幻觉核心保障）

### 1. 提示词策略

采用**系统提示词 + 任务提示词 + 动态注入**的三层架构，严格约束生成行为，杜绝幻觉：

1. 系统提示词（固定约束）

   

   明确角色为「资深合同合规法律专家」，核心规则：

   - 所有生成内容必须 100% 基于检索上下文，禁止使用模型自身参数知识，禁止编造法条、合同条款、法律意见；
   - 所有观点必须标注引用来源，格式为「依据《XX》第 X 条：xxx」；
   - 明确区分强制性规定与任意性规定，高亮风险点与法律后果；
   - 不确定内容必须明确说明「暂无相关有效法律依据」，禁止主观臆断。

2. 任务提示词（专项适配）

   

   针对合同审查、合规校验、条款比对、合同起草等不同任务，设计专属提示词，明确任务目标、输出格式、校验规则，保证输出标准化、专业化。

3. 动态提示词（实时注入）

   

   动态注入检索上下文、用户对话记忆、企业内部合规规则、用户画像信息，按需调整 Token 分配，压缩非核心内容，保证核心信息不丢失。

### 2. 生成策略

1. **闭卷约束生成**：强制模型仅使用检索到的上下文内容，禁用自身参数知识，从源头杜绝幻觉；
2. **低温度生成**：temperature 设置为 0.1-0.3，保证生成内容稳定、准确、不发散，适配合规场景的严谨性要求；
3. **分步生成 + 闭环校验**：复杂任务分步执行，每一步生成后都做事实性、合规性校验，校验不通过自动触发修正或补充检索；
4. **引用溯源机制**：生成内容的所有引用均可溯源至原始文档与条款，支持点击跳转核对，满足合规审计要求。

## 七、上下文管理与记忆策略

### 1. 上下文管理策略

1. **窗口分层管控**：将大模型上下文窗口固定拆分，严格控制各部分 Token 占比，例如 32K 窗口分配：系统提示词 2K、对话记忆 4K、检索上下文 16K、生成预留 10K，避免溢出；
2. **优先级动态压缩**：按法律效力优先级排序内容，高优先级内容完整保留，低优先级内容用小模型做摘要压缩，超阈值时自动丢弃最低优先级内容；
3. **长文档适配**：超长合同采用「核心信息锚定 + 分章节处理」策略，先提取合同核心定义、主体、标的、权责等锚点信息，再分章节检索生成，最后整合结果，避免上下文溢出。

### 2. 记忆策略

1. 短期记忆（对话上下文）
   - 滑动窗口管理最近 5-10 轮对话，超量内容自动生成对话摘要，保留核心意图；
   - 提取对话中的核心实体、用户需求，存入实体记忆，实现多轮对话上下文连贯。
2. 长期记忆（用户与合规知识库）
   - 用户画像记忆：存储用户所属行业、企业类型、常用合同类型、关注的合规领域，生成时自动注入，适配个性化需求；
   - 企业合规记忆：存储企业内部合规制度、审批流程、管理规定，作为长期检索知识库，保证生成内容符合企业内部要求；
   - 历史任务记忆：存储用户历史合同审查、合规校验结果，实现跨任务的信息复用。
3. **记忆安全**：不同用户、不同企业的记忆完全物理 + 逻辑隔离，严格控制访问权限，杜绝数据泄露。

## 八、Agent 框架设计

采用**主 Agent + 多子 Agent 的 ReAct 协同架构**，实现复杂合同合规任务的自动化拆解与执行。

1. 主 Agent（调度中枢）

   

   负责接收用户请求，做意图识别、任务拆解，调度子 Agent 与工具执行，监控执行状态，整合结果生成最终回复，采用 70B 级法律大模型，支持复杂任务调度与异常处理。

2. 专项子 Agent

   

   每个子 Agent 专注单一专项任务，配备专属提示词、工具与模型，保证专业性：

   - 文档解析 Agent：负责文档上传、解析、预处理、分块、元数据提取；
   - 检索 Agent：负责全流程检索、重排、校验、二次检索触发；
   - 合规审查 Agent：负责合同合规风险识别、法律依据匹配、修改建议生成；
   - 条款比对 Agent：负责合同版本比对、差异识别、冲突点分析；
   - 法律检索 Agent：负责法条精准检索、法律效力校验、司法解释匹配；
   - 合同起草 Agent：负责合规合同文本生成、模板匹配、条款优化；
   - 校验 Agent：负责全链路内容的事实性、合规性、引用溯源校验。

3. 工具系统

   

   模块化、可插拔的工具集，包括检索工具、文档处理工具、合规校验工具、计算工具、外部法条查询接口等，支持用户自定义工具扩展。

4. 执行与异常处理

   - 采用 ReAct 框架，执行「思考→行动→观察」循环，直到任务完成；
   - 子 Agent 执行失败自动重试，超阈值自动调整任务拆解方案；
   - 任务超时自动终止，支持断点续执行；
   - 无状态设计，支持水平扩展，异步队列削峰填谷，适配高并发。

## 九、高并发、高可用、高扩展性设计

### 1. 抗 1000QPS 高并发核心策略

1. 架构层
   - 全链路无状态微服务拆分：将预处理、嵌入、检索、推理、Agent 调度拆分为独立微服务，每个服务可独立水平扩展；
   - 读写全分离：所有数据库、向量库、图谱库均实现读写分离，只读节点水平扩展，支撑高并发查询；
   - 全链路异步化：非核心流程（入库、日志、记忆更新）全部异步化，不阻塞主请求链路。
2. 缓存层
   - 三级缓存架构：本地内存缓存（Caffeine）→分布式缓存（Redis Cluster）→向量库冷热缓存，高频请求直接命中缓存，降低后端压力；
   - 缓存预热：系统启动时提前预热高频法条、合规规范、常用合同模板的向量与内容；
   - 缓存防护：完善的防击穿、防雪崩、防穿透机制，保证缓存稳定性。
3. 流量控制层
   - 全链路限流：用 Sentinel 实现服务级、接口级限流，设置超阈值排队机制，避免服务被打挂；
   - 熔断降级：服务异常时自动熔断，降级返回缓存结果或友好提示，避免级联故障；
   - 削峰填谷：用 Kafka 实现请求队列化，匀速消费，应对突发高峰流量。
4. 推理优化层
   - 大模型用 vLLM 分布式部署，INT4 量化 + 连续批处理，单张 A100 可支撑 50+ QPS，20 张 A100 即可稳定支撑 1000QPS；
   - 嵌入模型、小模型用 ONNX Runtime 优化，批量推理，单张 T4 可支撑 1000+ QPS。

### 2. 高可用策略

1. **基础设施**：多可用区部署，硬件全冗余，避免单点故障；
2. **服务层**：所有服务多副本集群部署，K8s 容器编排，自动健康检查与故障转移，自动扩缩容；
3. **数据层**：所有存储多副本配置，全量 + 增量定时备份，异地容灾，RPO<5 分钟，RTO<30 分钟；
4. **发布层**：灰度发布 + 快速回滚机制，避免版本更新导致服务不可用。

### 3. 高扩展性策略

1. **架构层**：全模块插件化设计，嵌入模型、大模型、向量库、检索策略、Agent 工具均可插拔替换，无需修改核心代码；
2. **接口层**：标准化接口定义，新增模块只需实现标准接口即可接入；
3. **部署层**：云原生 K8s 部署，支持水平扩展，适配私有化、公有云、混合云部署；
4. **数据层**：分布式存储支持从几万到几千万文档的平滑扩展，增量更新无需全量重建索引；
5. **功能层**：支持自定义子 Agent、工具、提示词、合规规则，适配不同行业、不同企业的个性化需求。

## 十、可视化链路追踪与可观测性

基于 OpenTelemetry/Jaeger+Prometheus+Grafana 搭建全链路可观测体系，实现全流程可追溯、可排查、可审计。

1. 全链路追踪

   

   每个用户请求生成唯一 Trace ID，覆盖 Query 预处理、检索、重排、Agent 调度、大模型生成、结果返回全流程，每个环节记录 Span 信息（耗时、输入输出、状态、异常），存入 Elasticsearch，支持全链路回溯排查。

2. 可视化监控大盘

   - 系统监控大盘：实时展示 QPS、延迟、成功率、错误率、资源使用率、服务健康状态，异常自动告警；
   - 链路追踪大盘：输入 Trace ID 即可可视化查看全流程执行细节，定位问题根因；
   - 检索质量大盘：展示召回率、精确率、Top-K 命中率、各检索通道贡献度，支撑策略优化；
   - 大模型性能大盘：展示吞吐、QPS、延迟、Token 消耗、幻觉率，支撑模型优化；
   - Agent 执行大盘：展示任务成功率、执行步数、异常率，支撑 Agent 框架优化。

3. 用户侧溯源可视化

   - 生成结果的所有引用支持一键溯源至原始文档条款，展示原文内容；
   - 支持查看 Agent 执行全流程，包括任务拆解、工具调用、检索内容，提升用户信任度；
   - 全对话、全操作日志存档，支持导出，满足企业合规审计要求。

## 十一、召回质量评估体系

建立**离线自动化评估 + 在线业务指标评估**的双闭环评估体系，持续优化 RAG 效果。

### 1. 核心评估指标

表格







| 评估类型 | 核心指标            | 合规场景达标要求 |
| -------- | ------------------- | ---------------- |
| 离线评估 | Top10 召回率        | ≥95%             |
| 离线评估 | Top10 精确率        | ≥85%             |
| 离线评估 | MRR（平均倒数排名） | ≥0.9             |
| 离线评估 | NDCG@10             | ≥0.9             |
| 在线评估 | 用户采纳率          | ≥80%             |
| 在线评估 | 负反馈率            | ≤1%              |
| 在线评估 | 法务专家人工评分    | ≥8 分（10 分制） |

### 2. 测试集构建

由资深法务专家构建合同合规专业测试集，包含：

- 1000 + 条标准问答对，覆盖合同审查、法条检索、风险识别等全场景；
- 200 + 条难例集，覆盖模糊 Query、多意图 Query、边缘案例；
- 持续增量更新，将用户负反馈、新增场景加入测试集。

### 3. 持续优化闭环

1. **自动化评估流水线**：基于 CI/CD 搭建，每次版本更新自动触发离线评估，输出评估报告，不达标版本自动拦截；
2. **负反馈优化闭环**：用户负反馈自动进入优化队列，人工审核后加入测试集，针对性优化分块、检索、模型策略；
3. **A/B 测试机制**：新策略、新模型先做小流量 A/B 测试，指标优于旧版本再全量发布；
4. **定期模型微调**：用新增标注数据持续微调嵌入、重排、生成模型，提升法律领域适配性。

## 十二、数据安全与合规兜底

针对合同文档的涉密性，设计全链路数据安全体系，符合《数据安全法》《个人信息保护法》要求：

1. 全链路加密：传输用 TLS 1.3 加密，存储用 AES256 加密；
2. 细粒度权限管控：基于 RBAC 的角色权限体系，最小权限原则；
3. 企业级数据隔离：不同企业数据物理 + 逻辑双隔离，杜绝跨租户数据访问；
4. 全操作审计日志：不可篡改的操作审计记录，满足合规审计要求；
5. 敏感信息自动脱敏：身份证、银行卡、手机号等个人信息自动脱敏；
6. 数据生命周期管理：全流程管控文档的上传、存储、归档、销毁。
