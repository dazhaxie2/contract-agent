# 项目启动指南

合同合规 Agent 系统支持三种部署模式。**首次尝试推荐"模式 A: Docker Compose"，10 分钟就能跑起来**。

| 模式 | 适用场景 | 资源占用 | 启动时间 |
|---|---|---|---|
| **A. Docker Compose** | 开发自测、演示 | 内存 ~4 GiB | 5–10 分钟 |
| **B. Kubernetes**（Docker Desktop） | 模拟生产、验证 K8s manifest、走完整监控闭环 | 内存 ~6 GiB | 15–20 分钟（首次） |
| **C. 本地裸跑** | 单独调试 backend/frontend | 内存 ~2 GiB | 视依赖而定 |

> 模式 A 和 B 可以共存（端口不冲突），但同时跑会吃满 Docker Desktop 的 VM 内存，**建议二选一**。

---

## 模式 A：Docker Compose（推荐入门）

### 1. 启动全部服务

```powershell
docker compose up -d
```

首次启动需要拉镜像，约 3–8 分钟。监控全部 13 个容器健康：

```powershell
docker compose ps
```

### 2. 访问入口

| 服务 | 地址 | 说明 |
|---|---|---|
| 前端 | http://localhost:3000 | 登录后进入工作台 |
| 后端 API | http://localhost:8000 | Swagger: `/docs` |
| Grafana | http://localhost:3001 | admin / admin123 |
| Prometheus | http://localhost:9090 | scrape 状态 |
| Jaeger | http://localhost:16686 | 分布式追踪 |
| MinIO 控制台 | http://localhost:9001 | minioadmin / minioadmin123 |

### 3. 默认账号

首次启动后**没有用户**，需要先注册（首个用户自动成为 ADMIN）：

```powershell
curl.exe -X POST http://localhost:8000/api/v1/auth/register `
  -H "Content-Type: application/json" `
  -d '{\"username\":\"admin\",\"email\":\"admin@example.com\",\"password\":\"password123\",\"full_name\":\"Admin\"}'
```

之后浏览器登录 admin / password123 即可。

### 4. 单服务热更新

改了代码想看效果，**不必整套重启**：

```powershell
# 前端有 UI 改动 → 重建前端
docker compose up -d --build frontend

# 后端有代码改动 → backend 容器是 mount 模式，已开 --reload，保存即生效
# 若 requirements.txt 变了才需要重建：
docker compose up -d --build backend
```

### 5. 关闭 / 清理

```powershell
docker compose down            # 停容器但保留数据
docker compose down -v         # 彻底清理（含数据卷，慎用）
```

### 6. 模式 A 已修复的坑（写在前面，避免重踩）

- **Milvus 反复重启**：基础镜像必须用 `milvusdb/milvus:v2.5.4` + `command: ["milvus","run","standalone"]`；MinIO 凭证 env 必须叫 `MINIO_ACCESSKEYID` / `MINIO_SECRETACCESSKEY`（不是 `MINIO_ACCESS_KEY`）。
- **Swagger UI 空白**：FastAPI 的 SecurityHeadersMiddleware 默认 CSP 会拦掉 jsdelivr CDN，已在 `backend/app/middleware/security_headers.py` 给 `/docs /redoc /openapi.json` 开了豁免。
- **登录 502 Bad Gateway**：nginx 默认会缓存上游 IP，backend 容器重建后 IP 变了拦不回来。`frontend/nginx.conf` 已改为用 Docker 内置 DNS（`127.0.0.11`）+ 变量解析。
- **Nebula 集群"Machine not existed"**：v3 nebula storaged 不会自动注册到 metad，必须显式 `ADD HOSTS`。docker-compose 的 `nebula-init` 一次性容器会跑这步。
- **Postgres `sslmode` invalid kwarg**：asyncpg DSN **不接受** `?sslmode=...`，请勿在 `DB_DSN_*` 末尾加这个参数。

## 模式 B：Kubernetes（Docker Desktop）

把整套（backend / frontend / 7 个依赖 / 监控栈）跑到 K8s 里，**模拟最接近生产的环境**，并通过 kube-prometheus-stack + ServiceMonitor 自动采集。

### 1. 前置

- Docker Desktop 已启用 Kubernetes（Settings → Kubernetes → Enable Kubernetes → Apply & Restart）
- `kubectl` 能连：`kubectl get nodes` 看到 `docker-desktop Ready`
- Helm 已装：`helm version`
- 推荐给 Docker Desktop VM 调到 **≥ 8 GiB 内存**（Settings → Resources）

### 2. 一键部署 K8s 监控栈

```powershell
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo update
helm upgrade --install kps prometheus-community/kube-prometheus-stack `
  -n monitoring --create-namespace `
  -f k8s/monitoring/values.yaml --version "^65.0.0"

kubectl apply -f k8s/monitoring/servicemonitor.yaml
kubectl apply -f k8s/monitoring/prometheusrule.yaml
kubectl apply -f k8s/monitoring/grafana-dashboard.yaml
```

### 3. 一键部署 contract-agent

镜像复用本地 docker daemon（Docker Desktop 共享 image store），先确保已 build：

```powershell
docker compose build backend frontend
```

按依赖顺序 apply：

```powershell
kubectl apply -f k8s/local/00-base.yaml
kubectl apply -f k8s/local/10-data.yaml
kubectl apply -f k8s/local/20-milvus.yaml
kubectl apply -f k8s/local/30-kafka.yaml
kubectl apply -f k8s/local/40-nebula.yaml
kubectl apply -f k8s/local/50-app.yaml
```

等所有 Pod Ready（5–10 分钟）：

```powershell
kubectl -n contract-agent get pods -w
```

### 4. 创建 admin 用户（K8s 是全新数据库）

```powershell
kubectl -n contract-agent exec deployment/frontend -- sh -c `
  "wget --post-data='{\"username\":\"admin\",\"email\":\"admin@example.com\",\"password\":\"password123\",\"full_name\":\"Admin\",\"tenant_id\":\"default\"}' --header='Content-Type: application/json' -O- -q http://backend:8000/api/v1/auth/register"
```

### 5. 访问入口

| 服务 | 命令 / 地址 |
|---|---|
| 前端（NodePort） | http://localhost:30080 |
| 前端（port-forward） | `kubectl -n contract-agent port-forward svc/frontend 3002:80` → http://localhost:3002 |
| Backend Swagger | `kubectl -n contract-agent port-forward svc/backend 8001:8000` → http://localhost:8001/docs |
| Grafana | `kubectl -n monitoring port-forward svc/kps-grafana 3001:80` |
| Prometheus | `kubectl -n monitoring port-forward svc/kps-prometheus 9091:9090` |

Grafana 初始密码：

```powershell
$pw = kubectl -n monitoring get secret kps-grafana -o jsonpath='{.data.admin-password}'
[System.Text.Encoding]::UTF8.GetString([Convert]::FromBase64String($pw))
```

打开 Dashboards 应能看到 **Contract Agent / Overview**。

### 6. 卸载

```powershell
kubectl delete -f k8s/local/ --ignore-not-found
kubectl -n contract-agent delete pvc --all   # 清理数据卷
kubectl delete -f k8s/monitoring/ --ignore-not-found
helm uninstall kps -n monitoring
kubectl delete namespace monitoring contract-agent --ignore-not-found
```

### 7. 模式 B 已修复的坑

- **K8s 给所有 Pod 注入 `<SVC>_HOST` / `<SVC>_PORT=tcp://...` env**：把 backend pydantic 的 `REDIS_PORT` 整数字段和 cp-kafka 启动检查都打挂。`50-app.yaml` 和 `30-kafka.yaml` 都加了 `enableServiceLinks: false`。
- **Kafka broker 内部 controller 连不上自己**：K8s 里 `kafka` 默认解析到 Service ClusterIP（不是 Pod 自己），broker 启动时 controller 通过 advertised host `kafka:9092` 自连失败。`30-kafka.yaml` 用 `hostAliases` 把 `kafka` 指回 `127.0.0.1`。
- **node-exporter 启动失败**：Docker Desktop K8s 的根挂载不是 shared mount，hostPath `/` 失败；监控 `values.yaml` 已设 `nodeExporter.enabled: false`（真上 K8s 集群时恢复 true）。
- **Nebula init 死锁**：storaged healthcheck 依赖 ADD HOSTS，但 ADD HOSTS 又依赖 graphd healthy。`40-nebula.yaml` 把 storaged 改成 startupProbe（只看 TCP），让 init Job 单独跑。
- **登录 401 + 浏览器原地跳转**：K8s hidb 是空库，没有任何用户。第一次必须先调 `/auth/register`，参考上面第 4 步。
- **`passlib + bcrypt 4.x` 不兼容**：`bcrypt 4.x` 移除了 72-byte 兼容路径，passlib `detect_wrap_bug` 抛 `ValueError` 导致 register/login 全 500。`requirements.txt` 已 pin `bcrypt>=3.2,<4.0`。
- **Kafka Cluster ID 不匹配**：Kafka 持久化数据中的 Cluster ID 与 ZooKeeper 中的不匹配会导致 `InconsistentClusterIdException`。解决方案：删除旧数据卷 `kubectl -n contract-agent delete pvc kafka-data`，然后重新 apply `kubectl apply -f k8s/local/30-kafka.yaml`。

## 模式 C：本地裸跑（少用，主要用于断点调试 backend/frontend）

### 1. 只起依赖

```powershell
docker compose up -d hidb redis milvus minio kafka nebula-graphd
```

### 2. 跑后端

```powershell
cd backend
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
uvicorn app.main:app --reload --port 8000
```

环境变量按 `app/core/config.py` 默认值，本机要能解析 `hidb / redis / milvus / kafka / nebula-graphd` 这些 hostname——最简办法是给 hosts 文件加 `127.0.0.1 hidb redis milvus kafka nebula-graphd minio`。

### 3. 跑前端

```powershell
cd frontend
npm install
npm run dev    # http://localhost:5173，自动代理 /api 到 backend:8000
```
